Go module

From the docs: A collection of related Go packages. A single unit that defines the minimum requirements that must be satisfied by their dependencies.

  • a go.mod file is located in the root of your package. This location marks the module root, containing all packages in the root and subdirectories

    • note: subtrees can have their own go.mod files and will excluded

  • we used to have GOPATH, but instead we now have the concept of a main module

    • go commands will try to determine if the $PWD is in the module root (remember: this is where the go.mod file resides)? No, look up a directory, am I in the module root? No, look up a directory, am I .., etc.
    • being inside a module root or any of its subdirectories is referred to as being in module-aware mode

go.mod file

  • dependencies of your imported packages are with // indirect
  • if a dependency has a v2.0.0 or higher or and no go.mod file it’s suffixed with +incompatible (likewise for pseudo-versions, e.g., v2.0.1-0.yyyymmddhhmmss-abcdefabcdef+incompatible)

and example file:

module github.com/mfridman/srfax

require (
    github.com/mitchellh/mapstructure v0.0.0-20180715050151-f15292f7a699
    github.com/pkg/errors v0.7.0
)

exclude github.com/pkg/errors v0.7.1

GOPATH mode vs module-aware mode

In go1.11beta2, and 1.11, module support isn’t enabled by default and is controlled by the env variable GO111MODULE.

Default:

  • GO111MODULE=auto: go commands support modules outside the GOPATH (usually $HOME/go)

GOPATH mode:

  • GO111MODULE=off: disable go module support, as if you’re on 1.10

module-aware mode:

  • GO111MODULE=on: go commands will use go modules and never consult GOPATH to resolve imports

–

Commands

From here on we’re assuming module-aware mode, which you can achieve in one of two ways:

  1. copy/move/start a project outside your $GOPATH, yes you heard right, anywhere.
  2. set env variable to GO111MODULE=on. (option 1 is preferred)

Navigate to the root of your project and run ( go.mod cannot already exist):

go mod init

this will create a go.mod file.

If the project is using a third-party dependency manager, e.g., dep, godep, glide, etc., then go mo init will add requirements based on the existing configuration, e.g., Gopkg.lock.

Out of the box go commands will “just work”. There are no additional steps. Your favorite go commands build, install, run, test will just work.

As you run go commands the go.mod file will automatically be updated and new dependencies will be downloaded to satisfy imports.

Note the follow command:

go list

which also triggers the behaviour mentioned above (updating go.mod and downloading deps)
